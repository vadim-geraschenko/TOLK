# Background Video By Images Sequence Animation

Этот файл фиксирует актуальный рабочий подход к фоновой анимации `about.html` через image sequence.

## Основной файл

Текущий опорный вариант:

- [about.html](/home/rotvein/Desktop/TOLK_design/docs/design/pages/about/about.html)

Базовым и предпочтительным считать именно `about.html`.

## Когда использовать этот подход

Image sequence нужен, когда:

- фон должен быть `fixed` на всей странице;
- скролл должен управлять состоянием сцены;
- обычный `video.currentTime` scrub даёт рывки, запаздывания или скачки по keyframes;
- требуется предсказуемое визуальное поведение в браузере.

## Почему не `mp4` scrub

Для референсного ролика прямой scroll-scrub по видео оказался нестабилен:

- `mp4` сжат как обычное видео, а не как scrub-friendly ассет;
- keyframes недостаточно плотные для красивой scroll-перемотки;
- браузерный seek по `currentTime` хуже контролируется, чем покадровая подмена изображений.

Практический вывод:

- для этого кейса приоритетным считать image sequence, а не scroll-синхронизацию по `<video>`.

## Актуальная конфигурация sequence

- `60` кадров;
- формат: `webp`;
- `desktop`: ширина около `900px`;
- `mobile`: ширина около `600px`;
- отдельные наборы кадров для desktop и mobile;
- preload только локального окна вокруг текущего кадра.

Ассеты лежат в:

- `docs/design/pages/about/assets/angel-sequence/desktop`
- `docs/design/pages/about/assets/angel-sequence/mobile`

## Runtime-правила

Фон работает так:

- scroll position страницы переводится в нормализованный `progress`;
- `progress` маппится в индекс кадра;
- кадр подставляется в `fixed <img>`;
- preload идёт только в окне вокруг текущего кадра.

Текущее preload-правило:

- буфер `±6` кадров от текущего.

Это правило не менять без причины.

## Стартовая инициализация

При первой отрисовке важно не показывать `frame-001` по умолчанию.

Текущее правило:

- у `img` нет стартового `src` в HTML;
- страница сначала ждёт восстановление scroll position;
- затем вычисляет актуальный `progress`;
- первым видимым кадром должен становиться уже правильный frame для текущей позиции;
- фон становится видимым только после загрузки этого кадра.

Это защищает страницу от вспышки первого кадра при открытии или обновлении не с начала страницы.

## Маппинг прогресса

Для sequence используется не прямой `easeOutCubic`, а управляемый маппинг прогресса.

Текущее правило:

- анимация должна доходить до финальных кадров только у самого низа страницы;
- последние проценты скролла не должны быть “пустыми”, когда кадры уже закончились раньше.

Если меняется кривая прогресса, проверять:

- начало страницы;
- середину;
- последние `10–15%` scroll length.

## Responsive-правило

Выбор набора кадров:

- `desktop`, если ширина больше `760px`;
- `mobile`, если ширина `760px` и меньше.

Правило:

- не использовать desktop-sequence на mobile “про запас”;
- пользователь должен загружать только актуальный для его breakpoint набор.

## Текущее визуальное состояние базового варианта

В `about.html` сейчас допустимы только очень сдержанные атмосферные модификаторы:

- стартовое затемнение сцены;
- постепенное прояснение по мере скролла;
- лёгкая зернистость поверх;
- зернистость и затемнение сильнее в начале и слабее к концу страницы.

Что это означает на практике:

- фон в начале страницы более тёмный и чуть более “сырой” по фактуре;
- к концу страницы изображение становится чище и яснее;
- эффект должен поддерживать драматургию, а не превращать фон в спецэффект.

## Что не считать частью базового варианта

Для базового sequence-варианта не использовать как дефолт:

- тучи по бокам;
- локальные glow-эффекты вокруг свечи;
- canvas-рендеринг;
- staged-animation вместо sequence;
- сложные compositing-слои, если они не доказали практическую пользу.

Если такие эффекты когда-либо снова понадобятся, сначала отдельно проверить их пользу, а не встраивать по умолчанию в рабочий файл.

## Почему этот вариант предпочтителен сейчас

Плюсы текущего базового решения:

- визуально плавнее и надёжнее, чем прямой scrub по `mp4`;
- проще и дешевле в сопровождении, чем canvas-вариант;
- сохраняет контроль над scroll-driven сценой;
- не перегружает страницу лишней сценографией;
- лучше соответствует задаче `About`, где текст должен доминировать.

## Ограничения

- ассетов больше, чем у одного видеофайла;
- нужно контролировать суммарный вес;
- при увеличении числа кадров или разрешения вес растёт быстро;
- агрессивные дополнительные эффекты легко разрушают чистоту страницы.

## Production baseline

Если этот подход переносится дальше без пересмотра концепции, baseline должен оставаться таким:

- `60` кадров;
- `webp`;
- `900px` desktop;
- `600px` mobile;
- preload-буфер `6` кадров;
- fixed background через `<img>`;
- корректная начальная синхронизация с текущим scroll position;
- только сдержанное затемнение и зернистость как часть базового визуала.

## Что делать дальше

Если потребуется следующая итерация, развивать в таком порядке:

- сначала точная настройка кривой scroll progress;
- затем — тонкая настройка затемнения и зернистости;
- только после этого — любые дополнительные атмосферные слои.

Пока считать именно этот вариант актуальной рабочей опорой для image-sequence анимации на странице `About`.
